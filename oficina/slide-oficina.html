Oficina TV Box TX9 Pro

Descaracterização: do Android ao Armbian Linux

Passo a passo, comando a comando — conexão, extração e análise do DTB

Receita Federal IFSC Amlogic S905W

Apresentador: Romulo de Aguiar Beninca

Repositório do projeto: github.com/rbeninca/tvbox

Roteiro da Oficina

1. O Projeto

Origem dos dispositivos e objetivo da descaracterização.

2. Conexão

ADB: habilitar e conectar na TV Box.

3. Reconhecimento

Identificar SoC, Wi-Fi, partições e fazer backup mínimo.

4. Extração DTB

Baixar e descompilar o Device Tree.

5. Boot Linux

Preparar o SD, validar o boot e só então migrar para a eMMC.

Ao final, cada participante terá extraído o DTB do próprio dispositivo, salvo os backups mínimos e preparado o boot do Armbian.

Pré-requisitos: notebook Linux (ou WSL/macOS), cabo de rede + cabo USB-A/USB-A ou OTG, cartão microSD ≥ 8 GB, TV Box TX9 Pro.

O Projeto — Contexto

  • ▸Origem: TV Boxes apreendidos e destinados pela Receita Federal ao IFSC.
  • ▸Problema: firmware Android com processos root em segundo plano, sistema montado em rw e possíveis backdoors.
  • ▸Objetivo: trocar o Android por Armbian (Linux) mantendo o hardware funcional (display, Wi-Fi, IR).
  • ▸Resultado: dispositivo seguro, controlável e reaproveitado como nó Linux / mini-servidor.

Por que não só instalar um app sobre o Android? Uma camada de compatibilidade não elimina as vulnerabilidades nem o software malicioso do firmware original. A descaracterização precisa ir até o kernel.

O que é o DTB?
O Device Tree Blob é o "mapa" que diz ao kernel onde e como cada componente de hardware está ligado — endereços de memória, pinos GPIO e interrupções.
Por que ele é essencial?
Sem o DTB correto, o Linux pode até bootar, mas perde display, Wi-Fi e IR. Por isso a oficina foca em extraí-lo e entendê-lo.

1Conectar na TV Box (ADB)

O ADB (Android Debug Bridge) é a ponte para enviar comandos ao Android da TV Box. Como o aparelho já vem com root liberado, conseguimos ler partições diretamente.

1.1 — Instalar o ADB no notebook (Debian/Ubuntu):

sudo apt update
sudo apt install -y android-tools-adb android-tools-fastboot

1.2 — Habilitar a depuração na TV Box: Configurações → Sobre → tocar 7× em "Versão de compilação" → volta em Opções do desenvolvedor → ativar Depuração USB (e ADB via rede, se disponível).

Se a caixa não tiver tela/controle acessível, normalmente a porta de ADB over network (5555) já vem aberta — pule para o passo 1.4.

1Descobrir o IP da TV Box — ifrede.apk

Para conectar via rede (adb connect) você precisa saber o IP da TV Box. Como ela ainda está no Android e pode não ter tela acessível, use o aplicativo ifrede.apk — ele exibe o endereço de rede diretamente no display frontal da caixa.

Opção A — Instalar via navegador (na TV Box)

Se a TV Box tiver acesso à internet, abra o navegador Android e acesse:

https://rbeninca.github.io/tvbox/oficina/ifrede.apk

Baixe e instale o APK. Pode ser necessário habilitar "Fontes desconhecidas" nas configurações do Android.

Opção B — Instalar via cabo USB (com ADB)

Se já tiver o cabo USB conectado ao notebook:

# baixa o APK no notebook (uma unica vez)
wget https://rbeninca.github.io/tvbox/oficina/ifrede.apk

# instala na TV Box via ADB
adb install ifrede.apk
Depois de instalar, abra o app ifrede na TV Box. O IP aparecerá no display frontal (ex.: 192.168.1.50). Use esse endereço no próximo passo com adb connect.

1Conectar — comandos

1.3 — Conexão por cabo USB:

adb devices -l
adb shell id
adb shell su -c "id"

1.4 — Conexão por rede (ADB over TCP/IP):

# descubra o IP da TV Box no seu roteador (ex.: 192.168.1.50)
adb connect 192.168.1.50:5555
adb devices                       # agora aparece o IP:5555
adb shell                          # abre um shell dentro do Android
Nem todo shell do Android responde bem a whoami. Prefira id e, se preciso, rode os comandos críticos com su -c "...".

2Reconhecer o hardware

Antes de trocar o sistema, precisamos saber exatamente o que existe dentro da caixa. Cada comando responde uma pergunta:

# Qual o SoC / CPU? (esperado: Amlogic S905W, Cortex-A53, kernel 3.14.29)
adb shell cat /proc/cpuinfo

# Qual a plataforma? (ro.board.platform: gxl  ->  familia S905W/X)
adb shell getprop | grep -iE 'product.model|board.platform|hardware'

# Qual a versao REAL? (mascara: diz Android 10, mas...)
adb shell getprop ro.build.version.release   # -> 10  (falso)
adb shell uname -a                            # -> kernel 3.14.29
adb shell getprop ro.build.fingerprint        # -> 7.1.2/NHG47L (Android real)

Conclusão: kernel 3.14 é incompatível com Android 10 — o aparelho roda Android 7.1.2 mascarado.

2GPU, Wi-Fi e processos

# Processos com privilegio (ir_remote, vfd_display, etc.)
adb shell ps -A
adb shell mount        # revela sistema montado como "rw" (inseguro)

# GPU: modulo "mali" carregado? (S905W -> Mali-450 MP)
adb shell lsmod | grep -i mali
adb shell su -c "dmesg | grep -i mali"

# Wi-Fi: qual o chip? (procuramos ssv/rtl/wlan)
adb shell su -c "dmesg | grep -iE 'wifi|wlan0|ssv|rtl'"
adb shell lsmod | grep -i ssv      # -> modulo ssv6051 (chip SSV6051/6200)

2Backup mínimo antes de alterar a eMMC

Antes de qualquer boot Linux destrutivo, salve pelo menos boot, dtb, recovery e logo:

mkdir -p backup-tx9

adb shell su -c "dd if=/dev/block/boot of=/sdcard/boot.img"
adb shell su -c "dd if=/dev/block/recovery of=/sdcard/recovery.img"
adb shell su -c "dd if=/dev/block/logo of=/sdcard/logo.img"
adb shell su -c 'if [ -e /dev/block/dtb ]; then dd if=/dev/block/dtb of=/sdcard/android_tx9.dtb; else dd if=/dev/dtb of=/sdcard/android_tx9.dtb; fi'

adb pull /sdcard/boot.img backup-tx9/
adb pull /sdcard/recovery.img backup-tx9/
adb pull /sdcard/logo.img backup-tx9/
adb pull /sdcard/android_tx9.dtb backup-tx9/
Sem esse backup, o armbian-install e os ajustes de boot na eMMC tornam bem mais difícil voltar ao firmware original.

Por que o TX9 precisa de um DTB?

Num PC x86, o firmware UEFI/BIOS entrega tabelas ACPI ao kernel, e barramentos como PCI/PCIe permitem enumeração automática dos dispositivos. O sistema operacional "pergunta" ao hardware e monta o mapa sozinho.

Num SoC ARM como o Amlogic S905W, a maioria dos periféricos está ligada diretamente ao chip por endereços fixos de memória, GPIOs e IRQs — sem barramento autoidentificável. O kernel precisa receber essa descrição de fora: esse é o papel do Device Tree Blob.

PC x86TX9 / ARM SoC
Quem descreve o hardwareUEFI/BIOS + tabelas ACPI (DSDT, SSDT, FADT, MADT...)Device Tree Blob (.dtb)
Descoberta de dispositivosPCI/PCIe enumera automaticamentePeriféricos memory-mapped, sem enumeração
Intervenção do usuárioRaramente necessáriaÀs vezes precisa escolher/ajustar o .dtb
Em resumo: a tabela ACPI é para o PC x86 aquilo que o DTB é para placas ARM — um mapa que o kernel usa para entender a placa. No PC esse mapa vem embutido no firmware; na TX9, precisamos fornecê-lo manualmente ao trocar o Android pelo Armbian.

ACPI vs DTB — Visão Comparativa

Ambos fornecem ao kernel um mapa do hardware. No x86 isso vem via ACPI no firmware; em sistemas ARM embarcados, vem via Device Tree (DTB).

3O que é o DTB?

Device Tree Blob = binário que descreve a árvore de hardware: quais chips existem, em que endereço de memória, em quais pinos GPIO e interrupções.

  • .dtb — binário, entregue pelo bootloader ao kernel.
  • .dts — texto legível, obtido descompilando o .dtb.

No TX9, o .dts revelou o nó real do display (fd655_dev) e os pinos clk_pin/dat_pin. A partir daí correlacionamos isso com GPIOH_7/GPIOH_9 e com os bits 27/29 nos registradores do SoC.

fd655_dev {
  compatible = "amlogic,fd655_dev";
  clk_pin = <0x27 0x17 0x00>;
  dat_pin = <0x27 0x19 0x00>;
};

wifi {
  compatible = "amlogic, aml_wifi";
  interrupt_pin = <0x27 0x56 0x00>;
  power_on_pin = <0x27 0x55 0x00>;
};

Trechos reais do .dts do TX9. O nome SSV6051 veio do cruzamento com dmesg e lsmod, nao do nó do DTS.

3Extrair o DTB — passo a passo

3.1 — Tentativas que NÃO funcionaram (e por quê):

# Buscar assinatura do DTB (d00dfeed) dentro do boot.img -> nao concatenado
adb shell su -c "grep -c $'\xd0\x0d\xfe\xed' /dev/block/boot"   # 0
# Ler /sys/firmware/fdt  -> kernel 3.14 nega acesso

3.2 — Método que funcionou: ler direto o dispositivo de bloco do DTB:

# Copia o DTB que o bootloader entrega ao kernel para o cartao/sdcard
adb shell su -c 'if [ -e /dev/block/dtb ]; then dd if=/dev/block/dtb of=/sdcard/android_tx9.dtb; else dd if=/dev/dtb of=/sdcard/android_tx9.dtb; fi'

# Traz o arquivo para o notebook
adb pull /sdcard/android_tx9.dtb .

# Confere a assinatura magica (deve comecar com d0 0d fe ed)
xxd android_tx9.dtb | head -n 1
Em alguns firmwares o caminho é /dev/dtb em vez de /dev/block/dtb. Teste os dois.

3Descompilar e ler o DTB

3.3 — Instalar o compilador de device tree (dtc):

sudo apt install -y device-tree-compiler

3.4 — Converter binário (.dtb) em texto (.dts):

# -I dtb (entrada binaria)  -O dts (saida texto)
dtc -I dtb -O dts android_tx9.dtb -o extracted.dts

# Agora da para LER o hardware:
grep -iE 'fd655|aml_wifi|remote|GPIOAO_7|clk_pin|dat_pin' extracted.dts | head -n 40

3.5 — Extrair as partes úteis (os nós de cada periférico) para guiar a configuração do Armbian:

# Isola o no do Wi-Fi, do display e do IR em arquivos separados
sed -n '/wifi {/,/};/p'      extracted.dts > wifi.dts
sed -n '/fd655_dev {/,/};/p' extracted.dts > display.dts
sed -n '/rc@c8100580 {/,/};/p' extracted.dts > ir.dts

5Gravar no SD e bootar

5.1 — Descompactar e gravar no cartão microSD:

# Identifique o cartao (CUIDADO: /dev/sdX errado apaga seu HD!)
lsblk

# Grava a imagem (substitua sdX pelo seu cartao)
xz -dc Armbian_community_*_xfce_desktop.img.xz | sudo dd of=/dev/sdX bs=4M status=progress conv=fsync

5.2 — Ajustar a partição de boot para a base p281 usada no projeto:

sudo mkdir -p /mnt/boot
sudo mount /dev/sdX1 /mnt/boot

# troca o DTB padrao da imagem pelo base estavel do S905W
sudo sed -i 's/meson-gxl-s905x-p212.dtb/meson-gxl-s905w-p281.dtb/' /mnt/boot/extlinux/extlinux.conf

# compatibilidade com boot antigo de boxes Amlogic
sudo cp /mnt/boot/u-boot-s905x-s912 /mnt/boot/u-boot.ext
sudo cp /mnt/boot/dtb/amlogic/meson-gxl-s905w-p281.dtb /mnt/boot/dtb.img
sync
Algumas imagens usam /boot/armbianEnv.txt em vez de extlinux/extlinux.conf. No material testado do projeto, o fluxo reproduzível ficou em extlinux.conf + u-boot.ext + dtb.img.

5.3 — Insira o SD, ligue a TV Box (botão toothpick no conector AV) e valide o boot do Armbian antes de tocar na eMMC.

5Primeiro boot — validar a base Linux

Com o SD inserido, faça o assistente inicial do Armbian e valide a base antes de instalar qualquer coisa:

uname -r
findmnt /
lsblk -o NAME,SIZE,FSTYPE,MOUNTPOINT,LABEL
grep -nEi 'fdt|FDT|fdtfile' /boot/extlinux/extlinux.conf /boot/armbianEnv.txt 2>/dev/null
ip -br a
Antes da eMMC, confirme pelo menos: boot pelo SD, Ethernet funcional, acesso SSH e DTB base ajustado para p281.

5Primeiro boot — rodar o kit certo

O repositório separa duas funções: um script de pós-instalação da caixa e outro de manutenção do Wi-Fi.

# 1. va para a pasta em que voce copiou o kit
cd /caminho/do/TX9_PRO_SETUP_KIT
sudo chmod +x install_tx9.sh
sudo ./install_tx9.sh

# 2. so use o rebuild_wifi_v3.sh quando houver update de kernel
#    ou quando o modulo ssv6051.ko precisar ser recompilado
sudo bash /root/ssv6051-driver/rebuild_wifi_v3.sh
install_tx9.sh nao e a mesma coisa que rebuild_wifi_v3.sh: o primeiro prepara a caixa; o segundo recompila apenas o driver Wi-Fi.

5Migrar para a eMMC — armbian-install

Esta etapa grava o Armbian na memória interna, substituindo o Android de vez. Faça-a só depois de validar o sistema rodando pelo SD (SSH, Ethernet, serviços).

# ferramenta nativa do Armbian, executada como root
sudo armbian-install

No menu interativo, selecione:

[ Armbian install to internal media ]
  1 - Boot from eMMC - system on eMMC     <-- escolher 1 (sistema todo na eMMC)

[ Select filesystem type ]
  1 - ext4                                <-- escolher 1 (mais estavel)

O armbian-install copia o sistema do SD para a eMMC e prepara a estrutura de boot. Em muitos TX9 isso basta; em outros, ainda precisamos validar o bootloader da eMMC.

Operação destrutiva: apaga a eMMC (e o Android). Garanta que os backups de boot e dtb originais já foram salvos.

5Validar o boot da eMMC

No fluxo do projeto, o emmc_boot_setup.sh entra depois do armbian-install para conferir e corrigir o boot da memória interna:

# ainda bootado pelo SD, com a eMMC ja preenchida
sudo bash emmc_boot_setup.sh

# depois do reboot sem SD
lsblk
findmnt /
cat /proc/cmdline | grep -o 'root=[^ ]*'

O script checa extlinux.conf, root=LABEL=ROOT_EMMC, PARTITION_CONFIG=0x48 e a gravação do u-boot na eMMC.

Se a TV Box ainda iniciar apenas com o cartão inserido, este costuma ser o passo que está faltando.

Driver Wi-Fi SSV6051 — o problema

  • ▸O Wi-Fi usa o chip SSV6051/6200, sem suporte mainline no Linux moderno.
  • ▸Os drivers existentes são legados (kernel série 3.x); o Armbian usa kernel 6.x meson64.
  • ▸Não basta copiar o .ko do Android: é preciso recompilar out-of-tree contra os headers do kernel atual.
  • ▸Foi por isso que outros projetos de migração do TX9 perderam o Wi-Fi.
Os 3 obstáculos resolvidos:
  1. version magic — assinatura de versão do módulo não batia com o kernel ativo;
  2. firmware ausente — o chip precisa de .bin/.cfg em /lib/firmware;
  3. instabilidade — quedas por power management.

Código-base: driver parcialmente atualizado (a partir do repositório de Eloi Rotava), adaptado em ssv6051-driver/.

W1Corrigir o version magic

O Armbian nem sempre gera a string de versão completa que o kernel espera, causando erro ao carregar o módulo mesmo quando ele compila. A correção ajusta o EXTRAVERSION e recria o utsrelease.h para baterem com uname -r:

KVER=$(uname -r)
HDIR="/usr/src/linux-headers-$KVER"

# garante que os headers existem
sudo apt install -y linux-headers-current-meson64

cd "$HDIR"
# extrai o sufixo do kernel (ex.: "-current-meson64") direto do uname -r
EXTRAVERSION=$(echo $KVER | sed 's/[0-9.]*//')
sudo sed -i "s/^EXTRAVERSION =.*/EXTRAVERSION = $EXTRAVERSION/" Makefile
echo "#define UTS_RELEASE \"$KVER\"" | sudo tee include/generated/utsrelease.h
Detalhe importante do script real (rebuild_wifi_v3.sh): o sufixo não é fixo — é derivado do próprio uname -r com sed, então o procedimento sobrevive a atualizações de kernel.

W2Compilar o módulo ssv6051.ko

Com os headers corrigidos, compila-se o driver direto na TV Box, apontando o KBUILD para os headers do kernel ativo:

SRC_DIR="/root/ssv6051-driver/6051/ssv6xxx"
MOD_DEST="/lib/modules/$KVER/kernel/drivers/net/wireless/ssv6051"

cd "$SRC_DIR"
make clean || true
make KBUILD="$HDIR"          # gera ssv6051.ko contra o kernel atual

# instala o modulo e atualiza o indice de dependencias
sudo mkdir -p "$MOD_DEST"
sudo cp ssv6051.ko "$MOD_DEST/"
sudo depmod -a

Todo o fluxo (W1→W4) está automatizado em ssv6051-driver/rebuild_wifi_v3.sh — basta executá-lo na caixa.

W3Firmware e carregamento no boot

O chip carrega firmware auxiliar a partir de /lib/firmware. Sem esses arquivos o módulo até carrega, mas a interface não inicializa:

FW_DEST="/lib/firmware"
sudo cp "$SRC_DIR/ssv6051-wifi.cfg" "$FW_DEST/"
sudo cp "$SRC_DIR/ssv6051-sw.bin"   "$FW_DEST/"

# carrega o driver automaticamente a cada boot
echo 'ssv6051' | sudo tee /etc/modules-load.d/ssv6051.conf

No projeto, o arquivo ssv6051-wifi.cfg tambem desativa HT e AMPDU para reduzir quedas no barramento SDIO (erro -84 / brownout).

W4Estabilidade — desligar power save

Nos testes o Wi-Fi caía de forma intermitente: a causa era o gerenciamento de energia da interface. A solução é um hook do NetworkManager que desativa o power_save sempre que wlan0 ou wlan1 sobem:

sudo tee /etc/NetworkManager/dispatcher.d/99-ssv6051-fix >/dev/null <<'EOF'
#!/bin/sh
INTERFACE=$1
ACTION=$2
if [ "$ACTION" = "up" ] && { [ "$INTERFACE" = "wlan0" ] || [ "$INTERFACE" = "wlan1" ]; }; then
    /usr/sbin/iw dev "$INTERFACE" set power_save off
fi
EOF
sudo chmod +x /etc/NetworkManager/dispatcher.d/99-ssv6051-fix

# carrega e aplica agora
sudo modprobe ssv6051
IFACE=$(iw dev | awk '$1=="Interface"{print $2; exit}')
sudo iw dev "$IFACE" set power_save off

W5Conectar e validar

Conectar à rede com nmcli:

nmcli device wifi rescan
nmcli device wifi connect 'NOME_DA_REDE' password 'SENHA_AQUI'
nmcli connection modify 'NOME_DA_REDE' connection.autoconnect yes

Validação final — confirmar kernel, módulo, IP e power save:

IFACE=$(iw dev | awk '$1=="Interface"{print $2; exit}')
uname -r                        # kernel ativo esperado
lsmod | grep ssv6051            # modulo carregado?
echo "$IFACE"                   # wlan0 ou wlan1?
ip addr show "$IFACE"           # recebeu IP?
iw dev "$IFACE" get power_save  # deve estar "off"
Nao assuma o nome da interface. Em parte dos testes ela apareceu como wlan0; em outros, como wlan1.

DUsando o Display — exemplos

# Exibe texto fixo por 5 segundos (ate 4 caracteres)
tx9-display-show show_text "BOOT" --duration 5

DISPLAY_BRIGHTNESS=0x10 — brilho inicial
DISPLAY_BACKGROUND=clock_ip — tarefa de fundo (relogio + IP)

IRServico do Controle Remoto — tx9-ir

O daemon IR recebe eventos do receptor infravermelho (GPIOAO_7, driver meson-ir) e executa comandos shell mapeados para cada botao do controle.

Instalar o servico:

sudo bash install_ir_service.sh install

IRUsando o Controle Remoto — fluxo

Mapear um novo controle:

# 1. Escutar os scancodes do controle (aperte os botoes)
sudo tx9-ir-listen

# 2. Gerar o arquivo de configuracao interativamente
sudo tx9-ir-map /etc/tx9-ir/ir_daemon.conf

# 3. Editar os comandos no .conf (associar cada botao a uma acao)
sudo nano /etc/tx9-ir/ir_daemon.conf

# 4. Recarregar sem reiniciar o servico
sudo tx9-ir-reload

# 5. Acompanhar os logs ao vivo
sudo tx9-ir-log
Integracao IR + Display: cada botao pode disparar uma mensagem no display frontal. Exemplo no .conf:
0x40400b = tx9-display-show show_text UP --duration 1

O manual completo esta disponivel com tx9-ir-help.

Oficina Prática — Mãos à Obra

Agora vamos aplicar tudo em 4 passos diretos: gravar a imagem, bootar, instalar na eMMC e subir Docker.

1. Gravar

2. Bootar

Botão inferior + ligar

3. Instalar

Container rodando no TX9

Por que descaracterizar? O firmware Android original traz processos root desconhecidos, sistema montado em rw, versão mascarada e possíveis backdoors. A descaracterização substitui tudo por um Linux limpo e controlável.
O maior desafio técnico: o mapeamento do DTB. Cada TV Box tem periféricos (display, Wi-Fi, IR) ligados a pinos específicos do SoC que o kernel Linux não descobre sozinho. A imagem que vamos usar já traz o DTB correto para o TX9 Pro (S905W/P281), drivers compilados e serviços configurados.

1Gravar a imagem no cartão SD

1.1 — Baixar a imagem pronta (já inclui Armbian + DTB + drivers + serviços):

https://drive.google.com/file/d/1u4GrDhsN_l2zTHFpy2P6BQBlMFfIhFet/view?usp=sharing

Faça o download pelo navegador ou use gdown no terminal.

1.2 — Gravar no cartão usando o script da oficina:

# baixe o script de gravacao
wget https://rbeninca.github.io/tvbox/oficina/gravar-cartao.sh

# identifique o cartao SD (CUIDADO: confira o dispositivo!)
lsblk

# grave a imagem (substitua /dev/sdX pelo seu cartao)
sudo bash gravar-cartao.sh tx9-pro-armbian-reduzido.img /dev/sdX
Atenção: o script apaga todo o conteúdo do cartão. Confirme o dispositivo com lsblk antes. O script faz verificação automática (cmp) após a gravação.
Leitores USB lentos? Limite a velocidade para evitar erros:
RATE=10M sudo bash gravar-cartao.sh tx9-pro-armbian-reduzido.img /dev/sdX

2Bootar pelo cartão SD

  1. Desligue a TV Box (retire da tomada).
  2. Insira o cartão SD gravado.
  3. Com um palito ou clipe, pressione e segure o botão escondido no conector AV (parte inferior/traseira da caixa).
  4. Sem soltar o botão, ligue a TV Box na tomada.
  5. Segure por 5 segundos e solte. O Armbian vai iniciar pelo cartão.

3Instalar na eMMC — instal-aml.sh

Com o Armbian rodando pelo cartão SD, conecte-se via SSH ou use o terminal local e execute:

# acesse como root
ssh root@IP_DA_TVBOX       # senha: aluno123

# rode o script de instalacao
/root/instal-aml.sh

O script copia o sistema do cartão para a memória interna (eMMC), configura o bootloader e prepara tudo para funcionar sem o cartão.

Operação destrutiva: apaga o Android da eMMC permanentemente. Só execute após confirmar que o boot pelo cartão está funcionando.

Após a instalação:

  1. Desligue a TV Box
  2. Remova o cartão SD
  3. Ligue novamente — o Armbian deve iniciar pela eMMC

# exemplo: subir um servidor web
